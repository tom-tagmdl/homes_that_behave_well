# Concierge Signal Contract

## Purpose

Signals provide structured, stateful representations of household conditions and activities that are relevant to user interaction.

Signals enable Concierge to understand what is happening in the home and respond in a natural, context-aware way.

Signals are not owned by Concierge. They are exposed by integrations and orchestrated by Concierge.

---

## Definition

A Signal represents:

- A meaningful state of the home or household
- Derived and owned by an integration
- Expressed in a way that is understandable to a user

A Signal must not expose raw entity data.

A Signal must provide a normalized, human-consumable representation of system state.

---

## Examples

Signals include, but are not limited to:

- Calendar (upcoming events, current schedule)
- Shopping list (items, updates)
- Laundry (running, complete)
- Dishwasher (clean, dirty)
- Reminders and tasks
- Deliveries or arrivals

Each signal is owned and exposed by a responsible integration.

---

## Ownership Rules

Signals are always owned by an integration.

Examples:

- Calendar → calendar integration
- Shopping list → todo/list integration
- Laundry → appliance or sensor integration
- Dishwasher → appliance integration

Concierge must never:

- Derive signal meaning from raw entities
- Reconstruct signal logic independently
- Maintain its own copy of signal state

---

## Signal Structure

Each Signal must expose the following structure:

signal:
  id:                unique identifier
  type:              signal category (calendar, laundry, etc.)
  provider:          owning integration
  available:         boolean - is signal available
  state:             normalized state (integration-defined)
  summary:           concise human-readable description
  detail:            optional extended detail
  speakable:         fully formed natural language response
  actions:           optional list of supported actions

---

## State Requirements

Signal state must be:

- Deterministic
- Integration-defined
- Stable across requests

Examples:

laundry:
  state: complete

dishwasher:
  state: clean

calendar:
  state: upcoming_events

Concierge must not interpret or infer state beyond what is provided.

---

## Summary and Speakable Output

Each Signal must provide:

### Summary

- Short, human-readable
- Suitable for UI display

### Speakable

- Fully composed speech output
- Natural language
- No additional formatting required

Example:

Summary:
Laundry is complete

Speakable:
The laundry is done.

Concierge must not construct final phrasing from partial data.

---

## Actions

Signals may expose actions when appropriate.

Examples:

- Add to shopping list
- Remove item
- Mark task complete

Actions must be:

- Explicit
- Exposed via services
- Owned by the providing integration

Concierge may invoke actions but must not define action behavior.

---

## Availability

A Signal must declare whether it is available.

Reasons for unavailability may include:

- Integration not installed
- Required entities missing
- Configuration incomplete

Concierge must:

- Check availability before use
- Gracefully degrade if unavailable

---

## Service Interface Requirements

Signals must be accessible via defined services.

Minimum expected capabilities:

- Retrieve current signal state
- Retrieve summary and speakable output
- Invoke supported actions (if any)

Concierge must interact with signals only through these services.

---

## Eventing Model (Optional)

Signals may emit events when state changes.

Examples:

- laundry.complete
- dishwasher.clean
- shopping_list.updated
- calendar.event_starting

Events must be:

- Generated by the owning integration
- Consistent with the signal state model

Concierge may subscribe to these events for orchestration decisions.

---

## Context Integration

Signals are considered whole-home capabilities.

Concierge is responsible for:

- Determining when a signal is relevant
- Projecting signals into room context
- Selecting appropriate delivery method

Signals must not be tied to a specific room unless inherently location-based.

---

## Communication Rules

Concierge uses signals according to communication rules:

- Info → visual only
- Attention → optional voice
- Urgent → immediate voice

Signals do not determine communication level.

Concierge determines communication level based on:

- context
- priority
- user mode

---

## Failure Handling

If a signal is unavailable or incomplete:

Concierge must:

- Provide a clear response
- Avoid guessing or fabricating data
- Offer alternatives when possible

Example:

I don’t have access to laundry status right now.

---

## AI Usage

AI may be used to:

- Summarize signal outputs
- Assist with natural language responses

AI must not:

- Invent signal state
- Override integration-provided values
- Execute actions without explicit service calls

---

## System Behavior Rules

Signals must:

- Represent real system state
- Be explainable
- Remain consistent across UI and voice

Signals must never:

- Expose raw sensor values directly
- Require Concierge to interpret low-level data
- Create ambiguity in meaning

---

## V2 Governance Consumption

This contract consumes and aligns with:

- ADR-006 Capability Projection Governance Boundaries
- ADR-007 Experience Model Governance Boundaries
- ADR-009 Household Memory Governance Boundaries
- ADR-010 Household Productivity Experience Governance Boundaries
- ADR-011 Provenance Governance Boundaries
- ADR-012 Occupancy and Presence Governance Boundaries
- ADR-013 Concierge V1 Household-Facing Outcome Preservation Governance
- HACS and Platinum Governance Standard

This contract defines signal-consumption boundaries and does not redefine signal ownership.

---

## Ownership Boundary Validation

Ownership constraints in this contract:

- signal providers remain authoritative for signal state and lifecycle
- Concierge consumes signals for orchestration and communication only
- Coordinator V2 remains orchestration-only and does not become a signal source-of-record

---

## Terminology Alignment

Canonical terminology in this contract:

- Signal
- Context
- Capability
- Experience
- Provenance
- Occupancy
- Presence
- Scope

Legacy V1 assumptions that permit Concierge-side signal reinterpretation are prohibited.

---

## Final Principle

Signals represent the state of the household.

Concierge does not define that state.

Concierge reveals it, in the right place, at the right time, in the right way.
